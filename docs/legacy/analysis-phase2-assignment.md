## Second Phase of Game Development – Design

In this phase, the game concept is transformed into a concrete form.

We do not program, but:

* design how the game will look
* think through how individual systems will work
* prepare materials for development
* plan work and estimate time requirements

**Result:** a complete game design and an implementation plan.

---

## User Screens (Wireframes)

Design of the basic screen layout.

**Mandatory at minimum:**

* landing page
* login/registration
* main dashboard
* main activity screen (e.g., dragons, buildings)
* player interactions (battles, marketplace)
* leaderboard

They do not have to be visually polished, mainly clear. Rectangles with button and information labels are sufficient.

**Purpose:**

* think through the game flow
* prevent chaos during development

---

## User Flow

A diagram showing how the player moves through the game.

Example:

**Home → Registration → Dashboard → Training → Battle → Return to dashboard**

**Purpose:**

* understand the logic of player movement
* reveal missing screens

---

## Game Systems Details

Elaboration of systems from the concept:

* battles
* player progression
* breeding / crafting
* rewards
* time mechanics
* elements, etc.

Simple examples and rules may already be included.

**Purpose:**

* prepare clear rules for programming

---

## Game Economy (if applicable)

Tables:

* action costs
* rewards
* duration

**Purpose:** game balancing.

---

## Data Model (Database)

Simple ER diagram:

* main entities
* relationships between them

**Purpose:** data structure.

---

## API / Functions Design

List of the main system functions.

Example:

* registration
* dragon training
* battle
* creating a marketplace offer

**Purpose:** breaking the game into technical parts.

---

## Task Backlog

Work breakdown:

* task list
* division into sprints

Example:

* create login
* create dragon database
* create battles

**Purpose:** development planning.

---

## Workload Estimation (Person-Hours)

Students estimate the time required to complete each backlog task.

Example:

| Task        | Estimated Hours |
| ----------- | --------------- |
| Login       | 6               |
| Database    | 8               |
| Battles     | 8               |
| Marketplace | 10              |

They calculate the total project estimate.

**Purpose:**

* understand the scope of work
* realistically choose an MVP
* create a timeline

---

## What Does Not Belong Here Yet

* finished code
* full game graphics
* optimization
* marketing

---

## Outputs of the Second Phase

* wireframes
* user flow diagram
* system descriptions
* economic tables (if applicable)
* ER diagram
* backlog
* workload estimation
