**Assignment: Online Game**
Completion requirements

## Creating a Web Game

In teams, you will create a functional web game, deploy it to a server, and make it available to other students at the school. You will build upon the individual analysis and design that you have already processed.

### 1. Choosing a Scenario

* Present your proposals within the group.
* Agree on whose scenario you will implement.
* Consider the technical feasibility and scope of the project.

### 2. Technologies and Hosting

* Agree on the technologies used (frontend, backend, database, etc.).
* Consider what hosting you are able to secure and what it allows.
* The choice of technologies must correspond to real deployment.

### 3. Collaboration and Git

* You will collaborate via Git (e.g., GitHub/GitLab).
* Agree on working with branches, approving changes, and tracking tasks.
* Each member must contribute actively (own commits).
* Each team will submit a link to the Git repository where the project will be developed.
* The repository must be accessible to the teachers.
* Continuous work is expected - not a one-time upload of the finished project at the end.

### 4. MVP (Minimum Viable Product)

* First, you will create an MVP - a minimum functional version of the game.
* From the original preparation, base it on the breakdown: Must have, Should have, Could have.
* First, implement only the "Must have" items.
* The MVP must be functional and published on the server.

### 5. Planning Work and Responsibility

Before you start programming, you must think through the work, write it down, and divide it. The project must not be created randomly.

**Task List**
The team will create a shared task list (e.g., in issues or a project board). Each task must have a brief description, an expected outcome, an assigned author, and a status (planned / in progress / done). Work is done only on predefined tasks.

**Work Tracking**
Authorship must be traceable in the Git history. The project must contain a CHANGELOG.md file, where everyone records their completed tasks. In CHANGELOG.md, each student records the date, their name, and a brief description of the work done. Each student is responsible for their part of the project.

### 7. Deployment and Operation

You will not only create the game, but also actually deploy and operate it.

**Deployment**

* Publishing the application to a server.
* A functional public URL.
* Correct configuration of the production version.

**Operation**
Think about how the application behaves with a larger number of users, what happens in case of an error or invalid input, and how you solve potential outages or problems.

**Monitoring**

* Error logging.
* Tracking of important events (e.g., number of games, players).
* Application availability check.
* Deployment of the Google Analytics tool to track traffic and user behavior.

Part of the project is the ability to react to found errors and deploy a fixed version.
